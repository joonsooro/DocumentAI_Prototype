◊ Customer promise · verbatim from tight-thesis.json
An admin pastes messy commercial-invoice requirements and within one demo session sees compiled Document AI configuration, a clear capability assessment, a deterministic DAEJOO extraction, a readiness decision with business reasons, agentic clarification prompts for missed fields, admin-grade improvement recommendations grounded in observed corrections, and a separate internal flywheel view — all without telling the customer their requirement is "unsupported," without auto-lowering a threshold to fake a win, and without leaking internal roadmap gaps into any customer surface.
▲ Negative contract · verbatim from tight-thesis.json
Never display "unsupported" in the Customer Workspace; never recommend lowering a confidence threshold as the demo action; never surface raw prompts, raw model output, or roadmap-candidate signals (including the DAEJOO material-disposal phrase) on a customer surface; never call live OCR or live field extraction in v1; never mark a document Ready when any required field is below auto-confirm or when a workaround rule is unconfirmed for the supplier; never auto-promote a single correction into a ProductSignal without governance gating; never read the DAEJOO PDF from ~/Downloads at runtime (canonical path is app/assets/daejoo-invoice.pdf); never hide an AI Core agent failure behind a canned fallback (use ClarificationRequest + QualityMetric instead); never auto-generate a ProductSignal from the chat layer without explicit user consent (N9).

The Customer Workspace gets no Agent I/O Log entry point — the CustomerViewModel carries no field for agent payloads, so the mount is structurally impossible (preserves N1 + N3 by construction per A16).
