§ 01

Problem & Opportunity

Document AI administrators today translate prose business requirements into schema, prompts, regex, and thresholds by hand, producing silent capability mismatches that surface only as downstream STP failures or operator overrides. The prototype compiles messy customer intent into a single CompiledConfiguration artifact, surfaces capability assessments without ever telling the customer something is "unsupported," runs a deterministic DAEJOO extraction, renders readiness with business-language reasons, runs live AI Core agents for every reasoning step — compile, capability, readiness, clarification, admin recommendation, signal governance — and routes hidden capability gaps into a governed Internal Product Intelligence flywheel.

Field extraction is mock. Every agent is live.

§ 02

Customer Experience Contract

Customer promise. An admin pastes messy commercial-invoice requirements and within one demo session sees compiled Document AI configuration, a clear capability assessment, a deterministic DAEJOO extraction, a readiness decision with business reasons, agentic clarification prompts for missed fields, admin-grade improvement recommendations grounded in observed corrections, and a separate internal flywheel view — all without telling the customer their requirement is "unsupported," without auto-lowering a threshold to fake a win, and without leaking internal roadmap gaps into any customer surface.

Negative contract. Never display "unsupported" in the Customer Workspace; never recommend lowering a confidence threshold as the demo action; never surface raw prompts, raw model output, or roadmap-candidate signals (including the DAEJOO material-disposal phrase) on a customer surface; never call live OCR or live field extraction in v1; never mark a document Ready when any required field is below auto-confirm or when a workaround rule is unconfirmed for the supplier; never auto-promote a single correction into a ProductSignal without governance gating; never read the DAEJOO PDF from ~/Downloads at runtime (canonical path is app/assets/daejoo-invoice.pdf); never hide an AI Core agent failure behind a canned fallback — use ClarificationRequest + QualityMetric instead.

IDQuestionAssertion
HAPPY-1Who is the customer?The customer is always a Document AI administrator configuring commercial-invoice processing on behalf of an enterprise AP/logistics team; the end consumer of the configuration is always an internal operator triaging extraction results, never a generic end-user uploading invoices ad-hoc.
HAPPY-2What hurts today without this control layer?Administrators must today translate prose business requirements into Document AI schema, prompts, regex, and thresholds by hand, which always produces silent capability mismatches that surface only as downstream STP failures or operator overrides.
HAPPY-3How does the prototype solve the messy-intent → configuration problem?The prototype must compile customer intent into a CompiledConfiguration containing schema fields, field instructions, validation rules, regex patterns, confidence thresholds, and processing mode in a single visible artifact; the compile step is always a live AI Core agent call, never a static template lookup.
HAPPY-4How does the prototype handle a capability gap without lying?The Customer Workspace must classify every requirement as Supported or Supported with workaround and must never display Unsupported; capability gaps must always be routed to the hidden ProductSignal flywheel and to the customer-facing ClarificationRequest loop instead.
HAPPY-5What does the administrator do differently after running the demo?After the demo the administrator must review readiness in business language, confirm or correct extracted fields through the clarification loop, and accept or defer admin recommendations — they never edit raw prompts or thresholds blindly, and they always see why a recommendation was generated.
HAPPY-6What can a competing Document AI control layer not copy?Competitors always expose unsupported-capability language to the customer or auto-tune thresholds to hide failures; this prototype must instead split the surface into Customer / Admin / Internal workspaces with a governed flywheel, and that three-surface separation with a hidden product-signal pipeline is always the load-bearing differentiator.
EDGE-1What happens when extraction misses a required field?The system must always generate a ClarificationRequest asking the operator for field meaning, posting/review/reporting impact, and whether the interpretation applies to future documents from the same supplier; it must never silently fall back to a default value or skip the field.
EDGE-2What happens when an AI Core agent call fails, times out, or returns malformed JSON?An agent failure must always be surfaced as a ClarificationRequest with an operator-facing error message and must always be logged as a QualityMetric; the system must never substitute a canned fallback response that pretends the agent succeeded.
EDGE-3What happens when a correction looks like a one-off vs. a pattern?The flywheel must always gate CorrectionEvent → ProductSignal promotion through governance fields (frequency, customer impact, document type, supplier, country); a single one-off correction must never auto-promote into a roadmap recommendation.
EDGE-4What is the value of the prototype at demo scale and how does it scale?The demo must complete a full DAEJOO walkthrough in under five minutes, covering all three workspaces and all seven eval families; the same architecture must always scale to multiple suppliers and document types by adding deterministic fixtures and supplier-specific prompt versions, never by hand-editing the control layer.
RED-1What if a stakeholder asks the demo to lower thresholds to make a borderline document Ready for STP?The Admin Control Plane must always expose threshold management as a visible tool and must never include "lower the threshold" in any recommendation, suggestion, or demo script step, even if a document would pass STP after the change.
RED-2What if a stakeholder asks the demo to show the customer that a requirement (e.g. the DAEJOO material-disposal phrase) is unsupported, so the customer is "informed"?The Customer Workspace must never display the material-disposal phrase, any roadmap-candidate signal, raw internal prompt text, or any equivalent capability-gap language; that signal must always appear only in Internal Product Intelligence as unsupported_free_text_business_condition.
§ 04

Non-Goals & Scope Boundaries

v1 does NOT call live OCR or live field extraction (deterministic mock only for DAEJOO). v1 does NOT post to S/4HANA, does NOT deploy to SAP BTP, does NOT integrate with a real Document AI tenant. v1 does NOT introduce a database, does NOT introduce auth, does NOT introduce a shared hosted runtime. v1 does NOT support a second supplier beyond DAEJOO (Amazon stays a v2 happy-path fixture). v1 does NOT introduce 1–5 scoring or LLM-as-judge in evals (binary assertions only). v1 does NOT recommend threshold-lowering in any admin surface.

§ 06

Constraints, Dependencies & Risks

Stack constraints. React 18 + TypeScript + Vite + @ui5/webcomponents-react; in-memory typed fixtures only; Vitest binary evals. Live AI Core for all agentic steps (compile, capability, readiness reasoning, clarification, admin recs, signal governance).

Risk. Live AI Core agents are non-deterministic — evals assert agent behaviour invariants (shape, presence of fields, refusal patterns), not exact response text.

Risk. A single demo laptop with no shared runtime means a flaky network during a live demo blocks every agent; mitigation is the ClarificationRequest + QualityMetric surface per A5.

Dependency. AI Core API contract details (auth, endpoint, JSON shape) are still OPEN per OQ-1.

§ 08

Implementation Notes

Repo layout follows the brief: src/data/demoScenario.ts (fixtures), src/domain/types.ts (CustomerIntent, CompiledConfiguration, CapabilityAssessment, DocumentRun, ReadinessDecision, ExceptionReason, ClarificationRequest, CorrectionEvent, AdminRecommendation, PromptVersion, QualityMetric, ProductSignal, RegressionSignal, CapabilityGap), src/domain/harnessService.ts (pure service functions), src/evals/cases/*.json + src/evals/*.test.ts (Vitest binary evals).

Service functions: compileIntentToConfiguration, assessCapabilities, simulateDocumentRun (mock), decideReadiness, generateOperationalReasons, generateClarificationRequests, submitCorrection, generateAdminRecommendations, governProductSignals, detectRegressionSignals. Three workspaces are routed views over the same domain objects.

DAEJOO PDF lives at app/assets/daejoo-invoice.pdf and is referenced only by the static asset path; the ~/Downloads source is provenance only. AI Core failures surface as ClarificationRequest + QualityMetric (no canned fallbacks).
