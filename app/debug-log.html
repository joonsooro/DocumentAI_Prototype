Gulf tag (before fix)
Specification. The eval and the model both picked plausible reads of an ambiguous spec phrase — the spec did not pin a target field count, the prompt did not encode one, and the model satisfied the prose literally. The fix is to encode the spec author's decomposition at the prompt layer so the ambiguity disappears from the model's job. This is the "spec-encoding" form of Specification Gulf, distinct from "two equally valid model behaviours both correct" — here one was always intended to win, the prompt just didn't say so.
Why closed
The eval assertion now matches the live behaviour deterministically because the prompt encodes the 9-field rule explicitly. Re-running npm run evals:live three times in a row (not strictly part of the discipline, but cheap) would surface non-determinism if the prompt change is fragile; the one run we did took 45.8 s end-to-end and emitted the 9-field configuration through the zod schema cleanly. Regression check: mocked suite still 19/19 because mocks stub fetch and never read the live prompt. Fix entry resolves to closed.
Tenets surfaced
One durable tenet — added to CLAUDE.md §2: "When an eval pins a structural count (field count, item count, list length), the prompt must enumerate the items, not gesture at them. Gesturing leaves the count to the model's inference from prose, which is a Specification-Gulf trap; enumeration moves the count into the harness's encoded assumptions where it belongs." This survives a model swap — it is a how-we-build rule about the relationship between eval assertions and prompt content, not product content. Anthropic's framing: every component encodes an assumption the model can't do on its own; the prompt is the layer that encodes counts.
