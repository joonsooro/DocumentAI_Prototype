§ 01

Problem & Opportunity

Document AI administrators today translate prose business requirements into schema, prompts, regex, and thresholds by hand, producing silent capability mismatches that surface only as downstream STP failures or operator overrides. The prototype compiles messy customer intent into a single CompiledConfiguration artifact, surfaces capability assessments without ever telling the customer something is "unsupported," runs a deterministic DAEJOO extraction, renders readiness with business-language reasons, runs live AI Core agents for every reasoning step — compile, capability, readiness, clarification, admin recommendation, signal governance — and routes hidden capability gaps into a governed Internal Product Intelligence flywheel.

Field extraction is mock. Every agent is live.

§ 02

Customer Experience Contract

Customer promise. An admin pastes messy commercial-invoice requirements and within one demo session sees compiled Document AI configuration, a clear capability assessment, a deterministic DAEJOO extraction, a readiness decision with business reasons, agentic clarification prompts for missed fields, admin-grade improvement recommendations grounded in observed corrections, and a separate internal flywheel view — all without telling the customer their requirement is "unsupported," without auto-lowering a threshold to fake a win, and without leaking internal roadmap gaps into any customer surface.

Negative contract. Never display "unsupported" in the Customer Workspace; never recommend lowering a confidence threshold as the demo action; never surface raw prompts, raw model output, or roadmap-candidate signals (including the DAEJOO material-disposal phrase) on a customer surface; never call live OCR or live field extraction in v1; never mark a document Ready when any required field is below auto-confirm or when a workaround rule is unconfirmed for the supplier; never auto-promote a single correction into a ProductSignal without governance gating; never read the DAEJOO PDF from ~/Downloads at runtime (canonical path is app/assets/daejoo-invoice.pdf); never hide an AI Core agent failure behind a canned fallback — use ClarificationRequest + QualityMetric instead.

EDGE-1What happens when extraction misses a required field?The system must always generate a ClarificationRequest asking the operator for field meaning, posting/review/reporting impact, and whether the interpretation applies to future documents from the same supplier; it must never silently fall back to a default value or skip the field.
EDGE-2What happens when an AI Core agent call fails, times out, or returns malformed JSON?An agent failure must always be surfaced as a ClarificationRequest with an operator-facing error message and must always be logged as a QualityMetric; the system must never substitute a canned fallback response that pretends the agent succeeded.
EDGE-3What happens when a correction looks like a one-off vs. a pattern?The flywheel must always gate CorrectionEvent → ProductSignal promotion through governance fields (frequency, customer impact, document type, supplier, country); a single one-off correction must never auto-promote into a roadmap recommendation.
EDGE-4What is the value of the prototype at demo scale and how does it scale?The demo must complete a full DAEJOO walkthrough in under five minutes, covering all three workspaces and all seven eval families; the same architecture must always scale to multiple suppliers and document types by adding deterministic fixtures and supplier-specific prompt versions, never by hand-editing the control layer.

RED-2What if a stakeholder asks the demo to show the customer that a requirement (e.g. the DAEJOO material-disposal phrase) is unsupported, so the customer is "informed"?The Customer Workspace must never display the material-disposal phrase, any roadmap-candidate signal, raw internal prompt text, or any equivalent capability-gap language; that signal must always appear only in Internal Product Intelligence as unsupported_free_text_business_condition.
HAPPY-7How does the Internal workspace rank roadmap evidence so reviewers see the same order twice?The Internal Roadmap Evidence Ranking must always derive its score from a deterministic 5-factor linear formula over frequency, customerCount, workaroundBurden, actionability, and expectedStpLift (see A11 for the binding formula and app-spec.json#roadmap_ranking for constants); the same inputs always produce the same score, the same ordering, and the same reason line, and the panel must name the 5 factors verbatim in its closing info strip.
HAPPY-11How does the Customer Workspace clarification surface behave when intent is initially ambiguous?The Customer Workspace clarification surface must be a conversational chat thread; the user submits an intent, and from that point forward every clarification, re-compile announcement, capability-class question, and success summary flows through the chat (A12). When the per-turn chat.turn_decide agent returns action: 'recompile', F-04 must be re-invoked with the accumulated chat context appended to the original intent, and the Compiled Configuration, Capability Status, and Readiness panels must update in-place from the new result; clarifications that are pure missed-extraction must never surface a notify-team question (EDGE-7).
HAPPY-12What happens when the user asks for something the system cannot do (e.g. "fill these fields in S/4 HANA")?The chat.turn_decide agent must classify capability-class requests per the A14 v1 pattern list (integration · new document type · cross-document inference · predictive · bulk); at conversation end the assistant must surface the notify-team question, and only on explicit user 'yes' does a ProductSignal land in /internal with status: 'provisional' + provenance: 'conversational_notify_team' + "Being assessed for validity" tag (A13). The Customer Workspace must never tell the user the request is unsupported (N1 still binds).
EDGE-6What happens when the user declines the notify-team question at the end of a capability-class conversation?When the user replies 'no' to the notify-team question, the ConversationState must enter completed and no ProductSignal is created; the shared in-memory signal store must remain unchanged and the Internal RoadmapSignalsPanel must render no new row (N9 enforced).
EDGE-7Does every clarification conversation end with the notify-team question?No. When the chat.turn_decide classifier only ever emits missed-extraction actions (clarify / recompile) across the conversation, the assistant must never surface the notify-team question; the conversation transitions to success after re-compile + extraction completes, and no ProductSignal is created (A14, EDGE-7).
RED-3What if a code path tries to write a ProductSignal from the chat layer without surfacing the notify-team question first?The signal-write layer must reject any provisional-ProductSignal write that does not carry a matching ConversationState.status === 'awaiting_notify_decision' + explicit user 'yes' guard; N9 must be enforced at the data layer, not just at the chat UX layer. HAPPY-13How does an operator see what each agent received as input and what it returned as output during a session, so the system does not look like a black box?The prototype must expose an Agent I/O Log screen — reachable from a visible entry point on the Admin Control Plane and the Internal Product Intelligence workspaces — that renders one row per agent run in the current session (agent name, model, status, latency, token usage, redacted summary of input payload shape, redacted summary of output value shape, timestamp), sourced from the existing qualityMetricLog stream that is already mirrored to Langfuse per S4.5 OBSERVE-WIRE. § 04

Non-Goals & Scope Boundaries

v1 does NOT call live OCR or live field extraction (deterministic mock only for DAEJOO). v1 does NOT post to S/4HANA, does NOT deploy to SAP BTP, does NOT integrate with a real Document AI tenant. v1 does NOT introduce a database, does NOT introduce auth, does NOT introduce a shared hosted runtime. v1 does NOT support a second supplier beyond DAEJOO (Amazon stays a v2 happy-path fixture). v1 does NOT introduce 1–5 scoring or LLM-as-judge in evals (binary assertions only). v1 does NOT recommend threshold-lowering in any admin surface.

§ 06

Constraints, Dependencies & Risks

Stack constraints. React 18 + TypeScript + Vite + @ui5/webcomponents-react; in-memory typed fixtures only; Vitest binary evals. Live AI Core for all agentic steps (compile, capability, readiness reasoning, clarification, admin recs, signal governance).

Risk. Live AI Core agents are non-deterministic — evals assert agent behaviour invariants (shape, presence of fields, refusal patterns), not exact response text.

Risk. A single demo laptop with no shared runtime means a flaky network during a live demo blocks every agent; mitigation is the ClarificationRequest + QualityMetric surface per A5.

Dependency. AI Core API contract details (auth, endpoint, JSON shape) are still OPEN per OQ-1.

Entity extensions & new entities

ProductSignal (extended). Six new fields land on the existing entity:

  • customerCount: number ≥ 1 — distinct customers seeing the signal. v1 curated.
  • workaroundBurden: 'none' | 'low' | 'medium' | 'high' — v1 curated.
  • actionability: 'low' | 'medium' | 'high' — v1 curated.
  • expectedStpLift: number 0–50 — percentage points. v1 curated.
  • status: 'provisional' | 'governance_approved' — REQUIRED. Conversational signals enter as provisional; F-09 governance promotion sets governance_approved (A13).
  • provenance: 'governance_promotion' | 'conversational_notify_team' | 'curated_v1' — REQUIRED. The four curated-ranking fields above carry provenance: 'curated_v1' until S4 OBSERVE telemetry replaces them (OQ-5).

The 2-type ProductSignalType union (unsupported_free_text_business_condition, recurring_correction_pattern) stays. The reference design's additional labels (normalization_gap, parsing_edge_case, regex_too_strict, layout_ambiguity, schema_format_mismatch, workaround_repeated_correction) are illustrative-only display strings for the Internal Feedback Queue rendering; v1 routes all gaps through the existing 2-type taxonomy. Additional first-class types are deferred to S4 OBSERVE telemetry.

ChatTurn (new entity). A single turn in the Customer Workspace chat thread.

  • id: string
  • role: 'user' | 'assistant'
  • content: string
  • timestamp: ISO 8601 string
  • kind: 'message' | 'clarification_question' | 'recompile_announcement' | 'notify_team_question' | 'notify_team_confirmation' | 'success_summary'

  • id: string
  • turns: ChatTurn[]
  • compiledConfigVersionRefs: string[] — ordered list of CompiledConfiguration ids produced as the conversation progresses.

Agent I/O Log screen (HAPPY-13 · A16)

Purpose. Give the operator a coherent view of every agent run in the current session so the demo does not look like a black box. The screen is observability over agent traffic that is already being captured — qualityMetricLog (SUB-6) is the in-memory append-only store written by F-04, F-05, F-07, F-10, F-15, and (post-S4.5) chat.turn_decide, and is mirrored to Langfuse per S4.5 OBSERVE-WIRE. The Agent I/O Log renders that stream as a human-readable, session-scoped log; it does NOT introduce a new data source, a new agent, or a new persistence layer.

Audience & placement. The screen belongs to both the Admin Control Plane and the Internal Product Intelligence workspaces — its consumers are the admin operator and the internal product reviewer, both of whom legitimately see governance/visibility surfaces. It is NEVER reachable from the Customer Workspace (N1 + N3 by construction — the Customer view-model has no field carrying agent payloads). Entry point: a visible "Agent I/O Log" button in the workspace shell of /admin and /internal; the screen renders inline on the same route under data-testid="agent-io-log-panel".

What each row renders. One row per agent run in the current session, ordered newest-first: agent (e.g. aiCore.compile), model deployment id, status (success | fail), latency_ms, tokens_in / tokens_out, timestamp (ISO 8601), a redacted input-shape summary (field names + value types only — never the raw user/system prompt strings), a redacted output-shape summary (top-level keys of the parsed agent response — never the raw model text), and, for failure rows, the AgentFailure.reason enum + HTTP status. A row's expand affordance shows the same redacted shape summaries in a wider grid; there is no "show raw prompt" toggle in v1.

What the screen does NOT render. No raw system or user prompt text; no service-key strings; no substrings system:, prompt:, or <| (sanitiser pattern from A7 reused); no DAEJOO material-disposal phrase (N3); no ~/Downloads path (N8). The redaction is enforced at the row formatter, not at the F-18 store — the store already follows the agent_client_contract must_not list (prompts never enter QualityMetric), so the formatter just needs to defend against a future regression where a payload-shape summary inadvertently includes a prompt fragment.

Wiring. The screen reads from F-18's qualityMetricLog.subscribe() + getMetrics() (the same surface F-13's QualityMetricLogPanel already uses); rendering is reactive on every recordSuccess / recordFailure push. The Langfuse mirror is unchanged — the Agent I/O Log is the in-browser view of the same trace stream that S4.5 OBSERVE-WIRE pushes to Langfuse; the two surfaces stay in shape-parity but the screen does not call Langfuse at render time (privacy + offline-demo guarantee per SUB-6).
